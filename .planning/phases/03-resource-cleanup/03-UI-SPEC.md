## Interaction Patterns

### Exit Confirmation Dialog

**Trigger condition:** User attempts to leave page (close tab, navigate away, refresh) while `status === 'recording' || status === 'processing' || status === 'correcting'`

**Dialog type:** Browser native confirmation dialog (not custom modal)

**Dialog text:** Generic browser message (e.g., "Leave site?", "Changes you made may not be saved"). **Cannot be customized.**

**User actions:**

- **Stay:** Dialog cancelled, user remains on page, recording/processing continues
- **Leave:** Dialog confirmed, beforeunload cleanup executes, page unloads

**No dialog when:** `status === 'idle'` (D-02 from CONTEXT.md)

**Source:** D-01 and D-02 from 03-CONTEXT.md, RESEARCH.md beforeunload pattern (lines 379-393).

### Silent Cleanup

All resource cleanup is **invisible to the user** (no progress indicators, no confirmation messages). Cleanup includes:

1. Abort active fetch/SSE requests (AbortController)
2. Stop MediaRecorder
3. Stop all MediaStreamTracks (microphone LED turns off)
4. Close AudioContext
5. Close WebSocket
6. Clear timers

**Observable effects:**

- Microphone permission LED turns off immediately
- Browser Network tab shows cancelled requests (only visible in DevTools)

**No error messages:** Cleanup failures are logged to console only (not shown to user). Silent failure is acceptable because page is unloading anyway.

**Source:** D-03 from 03-CONTEXT.md, RESEARCH.md cleanup function pattern (lines 157-196).

## Design Decisions From Context

### Locked Decisions (from 03-CONTEXT.md)

Pre-populated from user decisions. These are NOT research recommendations — they are requirements.

1. **D-01:** Browser "Leave page?" confirmation dialog during recording/processing/correcting status
2. **D-02:** No dialog during idle status (user can leave freely)
3. **D-03:** Cleanup all audio resources (MediaRecorder, AudioContext, tracks, WebSocket) at page unload
4. **D-04:** Abort active fetch/SSE requests via AbortController
5. **D-05:** Move `stream` variable to component scope (from closure) for cleanup access
6. **D-06:** Consistent with Phase 1/2 error messaging (not applicable — Phase 3 has no user-facing messages)

### Inherited Patterns (from Phase 1 & 2)

7. **Glassmorphism design system:** Continued (no visual changes in Phase 3)
8. **Dutch language:** Continued (no text in Phase 3)
9. **Inline error display:** Continued (no errors in Phase 3)

### Claude's Discretion (from 03-CONTEXT.md)

These were open to research. Recommendations from RESEARCH.md:

- **AbortController pattern:** Use separate controller instances per request (`transcribeController`, `correctionController`) rather than single shared controller. Safer for independent cancellation.
- **beforeunload vs. pagehide:** Use both. beforeunload for desktop + confirmation dialog, pagehide for mobile fallback (more reliable when tab is killed without navigation).
- **Cleanup function structure:** Extract shared `cleanupResources()` function called from beforeunload, pagehide, and $effect cleanup. Prevents code duplication.
- **Cleanup order:** Abort requests first (fastest), then stop recorder (may trigger callbacks), then stop tracks, then close contexts, then close sockets, then clear timers. Order minimizes race conditions.

**Source:** RESEARCH.md Architecture Patterns (lines 94-283), Common Pitfalls (lines 312-370).
